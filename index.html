How Every Design Decision Was Made

Each UI element traces directly to the TDS/TCS spec document. Below is a simple explanation of what was built, why, and which section of the spec drove it.

1. Running TDS Estimate Banner
Spec §8A — "Running TDS estimate (between cycles)"
What it is: A small, light-colored banner below the wallet balance card showing the estimated TDS for the current 15-day cycle.

Why it's there: The spec says CSP should always know roughly how much TDS will be deducted at cycle end, so there's no surprise. It uses the word "अनुमानित" (estimated) clearly — it's not final until the cycle closes.

Design choices:
  • bgSubtle (#F1EDF7) background — spec says "light, non-intrusive indicator". We use the app's subtle background color (not white card, not yellow warning).
  • chip radius (8dp) — matches other inline banners in the app like BonusChip and InfoStrip.
  • ℹ icon + bodySmall text — keeps it quiet. The hero balance amount above must retain visual primacy (same principle as StateBanner).
  • Placed below wallet balance card — spec wireframe (§8A) shows it inside the wallet area, below "इस cycle की कमाई". User confirmed this placement.
Spec quote: "Show as a light, non-intrusive indicator — not a deduction entry. Use 'अनुमानित' (estimated) clearly — this is not final until cycle close. Updates as CSP earns more during the cycle. Disappears after cycle close and TDS is actually deducted."
2. Payout Summary Card (Expandable)
Spec §8B — "Payout summary card (at cycle close)"
What it is: A card that shows the complete earnings breakdown for the previous 15-day cycle. Collapsed by default — tap to expand and see the full gross → adjustments → TDS → net credited pipeline.

Why it's expandable: User asked for "same experience as security deposit and wallet card — clicking opens a detailed view." The Deposit and Bank cards in the app use the WalletCard wrapper with expand/collapse pattern (AnimatedVisibility with chevron rotation). We matched this pattern exactly.

Design choices:
  • WalletCard wrapper — white surface + 1dp border + 12dp radius + 16dp padding. Same as every other card on the wallet home.
  • Collapsed state: icon + title + amount + chevron — matches CardHeader pattern. Receipt (🧾) icon matches the tax context.
  • Net credited in green (statePositive #008043) — this is money that went INTO the wallet. Positive.
  • Expanded: four blocks — Gross Earnings → Operational Adjustments → Net Payable → Statutory Deduction → Net Credited. This exact ordering comes from spec §3 "Payout pipeline — the invariant".
Spec quote: "ONE card per cycle. Not per transaction. Clean. Readable. One deduction moment."
Spec §8C Rule 1: "NEVER show TDS as a separate wallet transaction entry. One card with breakdown."
3. Gross Earnings Always Shown First
Spec §8B + §8C Rule 3
What it is: Inside the expanded payout card, the first block is always "कुल कमाई (Gross)" showing base payout + bonus + recovery rewards.

Why: The spec is explicit — "Show GROSS first, always at the top. CSP must see what they earned before any deductions. Psychological anchor." And Rule 3: "NEVER deduct TDS before showing gross."

Order is: Base payout → Bonus → Recovery rewards → Subtotal → Adjustments → Net Payable → TDS → Net Credited. This mirrors the payout pipeline (§3) step by step.
4. TDS/TCS on Separate Lines with Section Reference
Spec §8B Table + §8C Rules 2 & 4
What it is: TDS appears as its own labeled line: "TDS (1% u/s 194C)" with the amount in red (stateNegative).

Why: Three spec rules drive this:
  • Rule 2: "NEVER use 'charge' or 'fee' or 'Wiom deduction' for TDS." We call it "वैधानिक कर कटौती" (statutory tax deduction) — it's government-mandated.
  • Rule 4: "NEVER hide TDS inside 'other adjustments'." Operational adjustments (carry fee, liability) are in a separate block from statutory deductions.
  • Section number visible: "CSP's CA will ask. Show it." That's why we show "(1% u/s 194C)" or "(20% u/s 206AA)".
If TCS also applies, it gets its own separate line below TDS.
5. Hindi Explainer Text
Spec §8B — "One-line explainer in Hindi"
What it is: At the bottom of the expanded payout card: "TDS सरकार को जमा होता है। यह आपकी income tax filing में credit मिलेगा।"

Why: The spec says this one line is critical — CSP must understand that TDS is not Wiom's charge, it goes to the government, and they get credit for it in their tax return. Without this, CSP may feel Wiom is taking their money.

Design: Uses metaXs (10sp) in textHint (#A7A1B2) — the smallest, lightest text style in the app. It's information, not a call to action.
6. Tax Summary Link Card
Spec §8D — "Annual tax summary (in Profile section)"
What it is: A simple card with a receipt icon + "टैक्स सारांश देखें" + chevron. Tapping opens the annual tax summary screen.

Why it's a card-style row: User feedback — originally this was a WiomNavRow with pink (brandPrimary) text. User said "text should be in black and same size of other card." So we switched to titleSm (16sp Bold) in textPrimary (#161021) with a textSecondary chevron, matching the CardHeader pattern.

Spec note: The spec says tax summary lives in "Profile section → Tax / Documents. Not in wallet feed." But for quick access during prototype testing, we added a link in the wallet home. The standalone Tax Summary screen (§8D) shows FY overview, Form 16A download, quarterly breakdown, and tax profile.
7. No-PAN Warning Strip
Spec Edge 3 (§9) — "CSP has no PAN"
What it is: A yellow warning strip inside the expanded payout card when PAN is missing: "PAN उपलब्ध नहीं — 20% TDS लागू। PAN जमा करें → TDS दर कम होगी"

Why: The spec says: "Section 206AA: TDS at 20% if PAN not furnished." And the CSP should see a prompt to submit PAN — "Strong incentive to provide PAN."

Design: Uses bgWarning (#FFF2BF) background with stateWarning (#B85C00) icon — matches the LiabilityStrip pattern used elsewhere in the wallet. The warning sits inside the TDS section of the expanded card, right after the TDS line showing 20%.
8. Below Threshold State
Spec §8E — "No-TDS state (below threshold)"
What it is: When annual cumulative payouts are below ₹30,000 (P_TDS_ANNUAL_THRESHOLD), TDS shows ₹0 with the message "सालाना सीमा से कम".

Why: The spec says: "CSP knows TDS exists but doesn't apply to them yet. No surprise when they eventually cross the threshold." The running estimate also shows ₹0 with "threshold से नीचे".

Design: No special styling — just regular bodySmall text. The full net payable is credited without any deduction. The payout card still shows the full breakdown so CSP sees their gross and adjustments.
9. All Cards Follow the Same Pattern
Design system consistency (WalletComponents.kt)
What it is: Every card on the wallet home screen uses the same WalletCard wrapper — white background (#FFFFFF), 1dp border (#E8E4F0), 12dp corners, 16dp padding.

Why: The wallet home has 5 cards (Balance, Recent Activity, Payout Summary, Deposit, Bank) + the Tax Summary link. User specifically asked for payout summary to "look like other cards like security deposit and wallet card." All cards use identical containers so the screen feels cohesive.

Design tokens used (from WiomTokens.kt):
  • bgSurface = #FFFFFF (card background)
  • strokePrimary = #E8E4F0 (card border)
  • WiomRadius.card = 12dp (corner radius)
  • WiomElevation.cardBorderWidth = 1dp (border width)
  • WiomSpacing.cardPadding = 16dp (inner padding)
10. No TDS in Transaction Entries
Spec §8C Rule 1 + Rule 5
What it is: TDS never appears as a separate line in the Recent Activity / Ledger section.

Why: Two rules:
  • Rule 1: "NEVER show TDS as a separate wallet transaction entry. '+₹3,600 payout' then '-₹72 TDS' as two entries = punishment feeling. One card with breakdown."
  • Rule 5: "NEVER show TDS on real-time credit entries. ₹35 credited. Period. No '-₹0.35 TDS.' TDS is cycle-level, not transaction-level."
How it works: Real-time credits show the gross amount. TDS is only visible in the cycle-end Payout Summary card as part of the full breakdown.

Payout Pipeline — The Invariant (Spec §3)

This ordering is non-negotiable. Every payout follows this sequence. The UI displays these steps in the same order inside the Payout Summary card.

1
Gross Earnings (Compensation OS)
Base payout + bonus + recovery rewards = gross_earnings
UI: Top section of payout card — "कुल कमाई (Gross)"
2
Operational Adjustments (Compensation OS)
Carry fee + FPV deductions + liability recovery = operational_adjustments
UI: Second section — "समायोजन" — shown in red (stateNegative)
3
Net Payable Determined
= gross_earnings - operational_adjustments
IF ≤ 0: no payout this cycle. No TDS. Deficit carries forward. STOP.
UI: Bold line — "शुद्ध देय राशि"
4
Threshold Check (Payment OS)
annual_cumulative + net_payable > ₹30,000?
IF NO: no TDS this cycle. Full amount credited.
UI: Shows "TDS: ₹0 (सालाना सीमा से कम)"
5
Statutory Withholding (Payment OS)
tds_amount = net_payable × tds_rate
tcs_amount = net_payable × tcs_rate (if applicable)
UI: "वैधानिक कर कटौती" section with TDS/TCS lines
6
Disbursement (Payment OS)
net_credited = net_payable - total_withholding → credited to wallet
UI: Bottom line — "आपके wallet में" in green
Spec §3: "Tax deduction NEVER comes before operational adjustments. If CSP earned ₹5,000 gross but ₹600 custody loss was recovered, taxable amount = ₹4,400. TDS on ₹4,400. Not on ₹5,000."

Design Invariants (Spec §8C) — What We Must NEVER Do

#RuleWhy
1 NEVER show TDS as a separate wallet transaction entry "+₹3,600 payout" then "-₹72 TDS" as two entries = punishment feeling. One card with breakdown.
2 NEVER use "charge" or "fee" or "Wiom deduction" for TDS It's statutory. Government mandates it. Wiom is the collector, not the beneficiary.
3 NEVER deduct TDS before showing gross Gross first. Always. CSP must see earnings before deductions.
4 NEVER hide TDS inside "other adjustments" Statutory deductions must be a separate, labeled line. Transparency.
5 NEVER show TDS on real-time credit entries ₹35 credited. Period. No "-₹0.35 TDS." TDS is cycle-level, not transaction-level.

Debug Panel Integration

How to test in the APK:

1. Open the wallet screen
2. Tap the 🐛 (bug) icon in the top-right corner of the header
3. Scroll to the "TDS / TCS Edge Cases" section
4. Tap any chip — the app writes the scenario key to MockStateController (SharedPreferences), loads the corresponding JSON variant, and navigates back to wallet home
5. The wallet reloads with the new data

How it works under the hood:
WalletDebugScreen → viewModel.applyWalletScenarioDebug("TDS_NORMAL") → debugBridge.set("walletState", "tds_normal") → MockStateController persists to SharedPreferences → next /wallet/home request → MockApiInterceptor reads snapshot → serves home__tds_normal.json
